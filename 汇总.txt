二、最终推荐
------------

如果目标是企业中文问答、客服、RAG、高并发推理服务：

推荐优先级：

1. vLLM
2. LMDeploy
3. SGLang

建议落地方式：

- 第一生产候选：vLLM TP=2
- 第二生产候选：LMDeploy TP=2
- 第三候选或特定场景候选：SGLang TP=2

不建议继续投入高并发生产测试：

- Transformers：原生基线，不适合作为高并发服务框架。
- ModelScope：生态集成有价值，但 transformers 后端吞吐太低。
- DeepSpeed：当前 wrapper 不适合 14B BF16 部署。
- Ollama：不面向企业高并发推理服务。


三、核心指标含义
----------------

1. requests
   - 本轮场景发出的请求总数。

2. success
   - 成功返回的请求数。

3. error_rate
   - 失败请求比例。
   - 计算方式大致为：失败请求数 / 总请求数。
   - 企业生产中应重点关注，通常应接近 0。

4. requests_per_s
   - 每秒完成的请求数。
   - 适合衡量服务能承载多少并发请求。

5. output_tokens_per_s
   - 每秒生成的输出 token 数。
   - 这是大模型推理吞吐最重要的指标之一。
   - 对聊天、RAG、客服这类输出型业务，通常比 requests/s 更关键。

6. total_tokens_per_s
   - 每秒处理的总 token 数。
   - 包括输入 token 和输出 token。
   - 长上下文场景中，这个指标能体现 prefill 能力。

7. latency_p50_ms / latency_p95_ms / latency_p99_ms
   - 端到端延迟分位数。
   - p50 表示一半请求低于该延迟。
   - p95 表示 95% 请求低于该延迟。
   - p99 表示 99% 请求低于该延迟。
   - 企业服务通常重点看 p95 和 p99，而不是平均值。

8. ttft_p50_ms / ttft_p95_ms / ttft_p99_ms
   - TTFT = Time To First Token，首 token 延迟。
   - 表示从请求发出到模型返回第一个 token 的时间。
   - 流式输出服务中，TTFT 直接影响用户感知速度。

9. output_tokens_mean
   - 平均每个请求实际生成的 token 数。
   - 本轮多数场景固定 max_tokens=256，因此该值接近 256 时说明输出长度比较一致。

10. json_valid_rate
    - 结构化 JSON 场景中，模型输出能被解析成合法 JSON 的比例。
    - 本轮 DeepSeek-R1-Distill 这类推理模型容易输出思考或解释文本，因此 JSON 有效率偏低不能直接代表服务框架失败。
    - 如果生产需要稳定 JSON，应单独测试 guided decoding、response format 或更强约束 prompt。

11. gpu_util_avg_pct / gpu_util_peak_pct
    - GPU 平均利用率和峰值利用率。
    - 利用率高不一定代表性能好，还要结合吞吐和延迟看。

12. memory_peak_mib
    - GPU 峰值显存占用。
    - 对双 5090 24G 来说，超过 22GB 就已经非常接近上限。

13. power_avg_w
    - GPU 平均功耗。
    - 可用于粗略估算 tokens/W。

14. temperature_peak_c
    - GPU 峰值温度。
    - 本轮 GPU0 多次达到 89C 到 92C，建议生产部署时关注散热和风道。

6. DeepSpeed
~~~~~~~~~~~~

结果目录：
/home/oem/llmDeploy/llm-benchmark-kit/results/deepspeed_tp2_nohup_8192fix

失败案例：
/home/oem/llmDeploy/llm-benchmark-kit/scripts/failure/deepspeed_tp2_14b_bf16_oom_failure.md

状态：
- 未进入正式压测。
- 启动阶段失败。
- 失败类型：CUDA OOM。

关键原因：

当前 DeepSpeed 服务封装的流程是：

1. transformers_openai_server.py 使用 AutoModelForCausalLM.from_pretrained 加载模型。
2. 加载时使用 device_map=auto。
3. device_map=auto 通过 Accelerate hooks 把 14B BF16 模型自动切分到双卡。
4. deepspeed_openai_server.py 再调用 deepspeed.init_inference。
5. DeepSpeed InferenceEngine 初始化时尝试 self.module.to(device)。
6. 这会移动一个已经由 Accelerate hooks 分片管理的模型。
7. GPU0 显存被打满，最终触发 OOM。

关键日志：

You shouldn't move a model that is dispatched using accelerate hooks.
torch.OutOfMemoryError: CUDA out of memory.

结论：

DeepSpeed 当前最小 wrapper 不适合部署 DeepSeek-R1-Distill-Qwen-14B BF16。该失败不代表双 5090 不能部署 14B，而是 DeepSpeed wrapper 的工程封装不适合当前场景。生产部署不建议继续投入 DeepSpeed。

八、生产部署建议
----------------

建议部署路线：

1. 先用 vLLM TP=2 上生产灰度。

   - OpenAI-compatible API 成熟。
   - 8192 长上下文已验证。

2. 保留 LMDeploy TP=2 作为备选。
   - 稳定性好。
   - 吞吐接近 vLLM。
   - 对 LMDeploy/TurboMind 生态和量化部署有价值。

3. 保留 SGLang TP=2 作为第三候选。
   - 稳定性好。
   - 如果后续需要 SGLang 的结构化、agent、workflow 编排能力，可以继续评估。

4. 不建议生产使用 Transformers / ModelScope transformers 后端做高并发。
   - 可以保留为原生框架基线或生态兼容性验证。

5. 不建议继续投入 DeepSpeed 当前 wrapper。
   - 若必须使用 DeepSpeed，需要重写多进程推理服务，不建议作为本轮选型必要工作。

6. 不建议将 Ollama 纳入企业高并发推理部署。
   - Ollama 更适合本地体验和轻量服务。
